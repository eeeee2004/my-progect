# Agent文档

## 酒店预订管理系统

---

## 一、项目理解层

### 1. Agent角色定义

你是本项目的代码实现者和文档维护者，是项目的建设参与者。你需要理解酒店预订管理系统的业务逻辑（三类用户角色、预订全流程、定价规则），基于现有代码进行增量开发和Bug修复。

### 2. Agent职责范围

- 代码实现：新增功能模块、修复Bug、优化已有代码。
- 文档维护：按照模板生成和更新14份项目文档。
- 测试验证：确保每次改动后 mvn test 全量通过。
- 数据库操作：谨慎执行 schema.sql 和 data.sql 的修改。

### 3. Agent禁止行为

- 禁止未经确认升级 Maven/npm 依赖版本。
- 禁止大规模重构现有代码（如改变目录结构、重写整个模块）。
- 禁止修改 git 配置或执行 `push --force` 等危险操作。
- 禁止提交包含密码、密钥等敏感信息的代码。
- 禁止在未确认前删除已有文件。
- 禁止跳过测试直接提交代码。

---

## 二、代码修改层

### 4. 修改原则

- **最小修改原则**：每次只改一个功能领域，不要同时动多个模块。
- **优先复用**：优先使用已有的工具类、异常类、注解，不要重复造轮子。
- **保持一致性**：新增代码风格（命名、异常处理、返回格式）必须与现有代码一致。

### 5. 代码修改规则

- 允许：增量添加新Controller/Service/Mapper/Entity。
- 允许：在现有方法中局部修改（如增加字段、调整逻辑）。
- 允许：新增 Vue 组件页面和路由。
- 禁止：改变项目目录结构（如移动包路径）。
- 禁止：重写已有的完整功能模块。
- 禁止：移除现有的安全配置或认证过滤器。

### 6. 文件管理规则

- 新建文件前评估是否可合并到已有文件中。
- 新建文件后需在提交信息中说明用途。
- 修改已有文件前先分析影响范围（谁引用了这个文件）。
- 删除文件前需确认不再被引用。

---

## 三、任务执行层

### 7. 提交与进度规范

- 每次只提交一个功能版本，提交信息格式：`vX.0-主题: 简短描述`。
- 提交信息的正文列出具体改动项，用减号开头。
- 完成后输出测试报告（通过数/失败数/错误数）。

### 8. 文档读取顺序

开发新功能前，必须按顺序了解：
1. 需求规格说明书（了解功能定义）
2. Agent文档（本文档，了解开发规范）
3. application.yml（了解当前运行配置）
4. 相关实体类和已有Controller/Service

### 9. 文档同步规则

- 新增功能后必须更新版本更新日志。
- 数据库表结构变更后必须同步更新 schema.sql。
- 种子数据变更后必须同步更新 data.sql（MERGE INTO 语法）。
- 所有文档更新后更新"最后更新"日期。

### 10. 风险与安全意识

- 启动后端前先清理 data 目录（H2环境）或确保MySQL连接正常（prod环境）。
- 不能在前端代码中硬编码任何密钥或密码。
- 所有密码必须通过 BCrypt 加密存储。
- 用户输入的手机号、身份证号等个人信息不出现在日志中。
- 上传文件的 Content-Type 和扩展名必须经过双重白名单校验。
